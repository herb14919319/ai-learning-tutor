# 第七章：生成式 AI 導入評估規劃

## 學習目標

完成本章後，學習者應能：

1. 說明企業導入生成式 AI 前應完成的需求與可行性評估。
2. 從問題、價值、資料、技術、成本、風險及組織條件評估應用情境。
3. 使用業務價值、導入難度與風險排列導入優先順序。
4. 區分概念驗證、最小可行產品、試點導入與正式推廣。
5. 說明總體擁有成本、投資報酬率與 Token 使用成本的基本概念。
6. 依資料敏感度、控制需求、效能與資源選擇部署方式。
7. 設計可衡量的關鍵績效指標與 PoC 成功／停止條件。
8. 辨識幻覺、隱私、公平性偏誤、提示注入、資料漂移及過度代理等風險。
9. 規劃人工監督（Human Oversight）、人工覆核、人工批准、權限、稽核、教育訓練與持續監控機制。

## 核心概念

生成式 AI 導入不是「先買工具，再找用途」，而是先確認問題是否值得解決、資料是否可用、技術是否可行、成本是否合理、風險是否可控，以及使用者與組織是否準備好。

完整導入可整理為以下流程：

> 需求盤點 → 場景排序 → 資料與風險評估 → 工具與部署選型 → PoC → MVP／試點 → 正式導入 → 持續監控與改善

每一階段都應有清楚的輸入、產出、負責人與決策條件。PoC 用來驗證假設，不是縮小版的正式上線；正式導入也不是專案終點，而是進入持續維運、監控與治理的新階段。

> **教材脈絡：**原始教材涵蓋導入流程、PoC／MVP、TCO／ROI、部署選擇、效能測試、風險管理與內容出處驗證。
>
> **補充說明：**教材曾將治理框架描述為正式導入後的工作。為避免誤解，本章調整為：PoC 階段就必須有與風險相稱的最低控制；當系統擴大使用時，再把治理制度完整化。
>
> **時效性更新：**商業工具版本、價格及個別產品功能未納入。安全框架、法規與組織指引可能更新，實際導入時應查閱適用地區與產業的最新官方文件。

## 詳細內容

### 7.1 導入評估的起點：先定義問題

導入團隊首先要把抽象期待改寫成可觀察的業務問題。例如「希望公司更智慧」無法直接評估；「客服平均需要十分鐘查找保固條款，希望縮短時間並維持回答正確」才是可分析的需求。

需求定義至少要回答：

1. 現在的流程與痛點是什麼？
2. 受影響的使用者與利害關係人是誰？
3. 問題造成多少時間、成本、錯誤或服務影響？
4. 為什麼考慮生成式 AI，而不是流程改善、搜尋、規則系統或其他方法？
5. 預期成果如何衡量？
6. 哪些事情明確不在導入範圍內？

若問題不明確、沒有資料或無法定義成功條件，應先完成需求盤點，而不是直接比較模型或採購工具。

### 7.2 七大評估面向

| 評估面向 | 關鍵問題 | 可接受的證據 |
| --- | --- | --- |
| 問題明確性 | 痛點、流程與使用者是否清楚？ | 流程圖、訪談紀錄、基準數據 |
| 業務價值 | 能否提升效率、品質、收入或服務？ | 可衡量效益與優先級 |
| 資料條件 | 資料是否足夠、正確、可用且有權使用？ | 資料清單、品質檢查、權限依據 |
| 技術可行性 | 現有工具、RAG、API 或自建方案能否完成？ | 原型測試、整合與效能結果 |
| 成本效益 | 全生命週期投入是否小於合理預期效益？ | TCO、ROI、敏感性分析 |
| 風險程度 | 錯誤、外洩、偏誤或越權的影響多大？ | 風險清單、控制措施、殘餘風險 |
| 組織接受度 | 使用者、主管與維運團隊是否能採用？ | 訓練計畫、試用回饋、角色分工 |

這七個面向要一起看。技術上能做，不代表資料可以使用、成本划算或組織能承擔風險。

### 7.3 應用場景的優先順序

教材建議以「業務價值」與「導入難度」建立四象限。實務上還應加上風險，避免高價值、高風險情境因效益誘人而被過早自動化。

| 場景 | 建議策略 |
| --- | --- |
| 高價值、低難度、低風險 | 優先安排 PoC，快速驗證成效 |
| 高價值、高難度或中高風險 | 先做資料、技術與控制設計，再進行受限 PoC |
| 低價值、低難度、低風險 | 可作為訓練或探索案例，但不應排擠高價值需求 |
| 低價值、高難度或高風險 | 暫緩、改寫需求或尋找非 AI 解法 |

排序時應使用一致標準，例如價值、資料準備度、技術成熟度、整合難度、風險、成本及使用者接受度。分數只是討論工具，不能取代證據與責任判斷。

### 7.4 資料可用性與治理評估

生成式 AI 專案常在資料階段遇到阻礙。資料評估應確認：

- **來源**：資料來自何處，是否能追溯？
- **品質**：內容是否正確、完整、一致且仍然有效？
- **權限**：組織是否有權將資料用於此目的？
- **敏感性**：是否含個資、機密、合約或受限制資訊？
- **代表性**：資料是否只反映少數情境或特定族群？
- **更新機制**：政策、產品或文件變更時如何同步？
- **刪除與保存**：資料何時刪除，備份與日誌如何管理？

資料「存在」不等於「可用」。如果文件版本混亂、權限未切分或資料用途不明，直接建立知識庫可能放大錯誤與外洩風險。

### 7.5 工具與技術方案選型

選型不應只比較模型回答好不好，還要評估完整系統能力：

第六章從任務、模態、輸入輸出與單次使用風險選擇工具類型；本章則從組織導入角度，主責資料條款、整合、成本、供應商、部署、稽核與退出能力的方案選型。

| 評估項目 | 應確認事項 |
| --- | --- |
| 功能符合度 | 是否完成核心任務與必要語言／模態？ |
| 資料處理 | 輸入是否被保存、再利用或傳送到其他地區？ |
| 整合能力 | 是否支援所需 API、身分驗證、資料庫或工作流程？ |
| 權限與稽核 | 能否依角色授權並保留必要紀錄？ |
| 可用性 | 實際使用者能否正確且穩定操作？ |
| 效能與可靠性 | 延遲、容量、可用性與故障處理是否符合需求？ |
| 成本 | 授權、用量、整合、資料與維運成本是否合理？ |
| 可移轉性 | 若供應商、模型或需求改變，資料與流程能否調整？ |
| 供應鏈風險 | 第三方模型、外掛、連接器與套件如何審查？ |

工具功能與商業方案會改變，因此正式選型必須用當下的官方文件、合約及實測結果確認，不能只依教材中的品牌清單。

### 7.6 TCO、ROI 與 Token Economics

#### 7.6.1 總體擁有成本

總體擁有成本（Total Cost of Ownership, TCO）是系統在評估期間內的全部成本，不只包含模型或 API 費用。

常見成本包括：

- 工具、授權、API 與運算資源。
- 需求訪談、PoC、開發、測試及系統整合。
- 文件整理、資料清洗、標註、去識別化與更新。
- 教育訓練、使用支援與變更管理。
- 權限、監控、資安、法遵與稽核。
- 維運、故障處理、版本調整及退出成本。
- 錯誤輸出、資料外洩或服務中斷造成的風險成本。

#### 7.6.2 投資報酬率

投資報酬率（Return on Investment, ROI）的基本概念為：

> ROI =（效益 − 成本）÷ 成本 × 100%

效益可能包括節省工時、降低錯誤、減少重工、改善服務品質或增加收入。無法直接換算金額的效益，可以另外列為品質、滿意度或風險指標，不應任意轉成誇大的財務數字。

回收期（Payback Period）與 ROI 不同。回收期關注多久收回初始投入；ROI 關注投入相對產生多少淨效益。計算時應明確寫出期間、假設與資料來源。

#### 7.6.3 Token Economics

Token Economics 在本教材脈絡中，主要指推論階段輸入與輸出 Token 的使用量及相關成本。規劃時可考慮：

- 提示、文件與歷史對話的輸入量。
- 模型輸出的長度與呼叫次數。
- RAG 檢索片段是否過多。
- 是否可使用摘要、快取、批次或較適合的模型降低浪費。

Token 成本只是 TCO 的一部分，不能用它代替整體成本評估。

### 7.7 部署方式選擇

| 部署方式 | 主要優點 | 主要限制 | 常見評估情境 |
| --- | --- | --- | --- |
| 雲端服務／API | 建置快、資源彈性高 | 資料處理、供應商依賴與用量成本需確認 | 快速 PoC、一般生產力應用 |
| 自建或私有環境 | 資料與系統控制程度較高 | 需承擔基礎設施、人才、維運與更新 | 敏感資料、特殊控制或整合需求 |
| 邊緣部署 | 可就近處理、降低網路依賴 | 裝置算力、記憶體、更新與監控受限 | 產線、現場設備、離線或低延遲需求 |
| 混合部署 | 可依資料與工作負載分流 | 架構、身分、資料同步與治理較複雜 | 同時存在敏感資料與彈性需求 |

私有化部署不會自動消除幻覺、公平性偏誤或設定錯誤；雲端服務也不必然不安全。應依資料流向、合約、控制能力、技術資源與實測結果判斷，而不是用單一標籤決定。

### 7.8 PoC、MVP、試點與正式導入

| 階段 | 主要目的 | 典型範圍 | 決策問題 |
| --- | --- | --- | --- |
| PoC | 驗證技術可行性、業務價值與主要風險 | 小資料集、少數任務、受控使用者 | 這個構想是否值得繼續？ |
| MVP | 建立具核心功能的最小可用版本 | 可完成主要使用流程 | 使用者是否真的會用且能產生價值？ |
| 試點 | 在受限真實環境測試營運 | 單一部門、有限案件與監控 | 流程、支援與控制能否運作？ |
| 正式導入 | 擴大到核准範圍並持續治理 | 多使用者、正式資料與服務承諾 | 能否穩定、安全且可維護地運作？ |

四者不可混為一談。PoC 成功不等於可以直接全面上線；還需要驗證整合、使用者行為、例外處理、權限、維運與治理。

### 7.9 PoC 的設計要素

一份可判斷的 PoC 應包含：

1. **測試目標**：要驗證什麼假設？
2. **範圍與排除項目**：哪些資料、使用者及情境納入或排除？
3. **基準線**：未使用 AI 時的時間、品質與成本為何？
4. **測試資料**：資料來源、版本、品質與權限是否清楚？
5. **成功指標**：效率、正確性、採用率與風險門檻為何？
6. **控制措施**：如何去識別化、限制權限、人工覆核及記錄結果？
7. **時程與角色**：誰負責業務、資料、技術、安全與驗收？
8. **退出條件**：未達標、風險過高或成本失控時如何停止？

例如，客服 FAQ PoC 可以限定在已核准的低風險問題，只產生回覆草稿，不處理退款、法律爭議或帳戶異動；由客服人員確認後才可送出。

### 7.10 KPI 與成效衡量

關鍵績效指標（Key Performance Indicator, KPI）要能對應最初問題，並同時觀察效益與風險。

| KPI 類型 | 指標例子 | 注意事項 |
| --- | --- | --- |
| 效率 | 平均處理時間、回覆時間、每小時產量 | 應與導入前基準比較 |
| 品質 | 正確率、錯誤率、退件率、使用者滿意度 | 定義何謂正確及由誰判定 |
| 成本 | 單件成本、重工成本、用量與維運成本 | 不只計算 API 費用 |
| 採用 | 活躍使用者、完成率、持續使用率 | 使用次數高不等於品質好 |
| 風險 | 幻覺、機密外洩、越權或違規事件 | 應設定不可接受門檻 |
| 學習 | 訓練完成率、錯誤類型改善、模板採用率 | 用來評估組織準備度 |

KPI 應包含基準值、目標值、測量期間、資料來源與負責人。若只設定「使用人數」，可能鼓勵大量使用卻忽略品質與風險。

### 7.11 效能測試：Latency 與 Throughput

延遲（Latency）是單次請求從送出到取得回應所需的時間；吞吐量（Throughput）是單位時間內可處理的工作量。

- 即時客服或互動介面通常重視低延遲。
- 大量離線摘要、分類或內容產生通常更重視吞吐量。
- 批次處理可能提高整體吞吐量，但等待成批可能增加單筆延遲。

效能測試還應包含並行使用、尖峰負載、長時間穩定性、外部服務失敗、重試與降級處理。只測一次成功回應，不能代表正式環境可靠。

### 7.12 生成式 AI 風險盤點

| 風險 | 典型現象 | 控制措施 |
| --- | --- | --- |
| 幻覺（Hallucination） | 產生錯誤事實、數字或來源 | RAG、來源限制、查核與拒答條件 |
| 資料隱私 | 輸入或輸出洩漏個人或敏感資料 | 資料最小化、去識別化、權限與保存控制 |
| 公平性偏誤（Fairness Bias） | 資料、設計或使用情境不平衡，使不同群體受到不合理差異或刻板化影響 | 分群測試、公平性檢查、人工覆核與改善 |
| 著作權與授權 | 來源或生成內容的使用權不明 | 保存來源、授權檢查、必要時專業審查 |
| 提示注入（Prompt Injection） | 惡意內容誘使系統忽略規則或洩密 | 信任邊界、輸入檢查、權限隔離、輸出驗證 |
| 不安全輸出 | AI 輸出被下游系統直接執行 | 型別與規則驗證、沙箱、禁止未審核高風險動作 |
| 過度代理（Excessive Agency） | Agent 權限過大或自行執行不可逆動作 | 最小權限、動作白名單、人工確認、稽核 |
| 資料漂移（Data Drift） | 上線後輸入分布改變，品質逐步下降 | 持續監控、告警、重新評估與必要更新 |
| 供應鏈風險 | 第三方模型、連接器或套件出現問題 | 供應商審查、版本管理、權限限制與退出計畫 |
| 過度依賴 | 使用者未查證就採用結果 | 教育訓練、責任分工、人工決策與抽查 |

資料漂移發生在部署後環境或輸入特性改變；過度擬合則是模型在訓練階段未能良好泛化。兩者發生階段與處理方式不同。

### 7.13 風險處理策略

組織可依風險發生可能性與影響，選擇下列策略：

- **降低（Mitigate）**：加入控制措施後繼續使用，例如對外回覆需主管覆核。
- **迴避（Avoid）**：停止或不進行無法接受的高風險用途。
- **轉移（Transfer）**：透過合約、保險或供應商責任分擔部分風險，但組織不能因此免除自身責任。
- **接受（Accept）**：在理解並記錄殘餘風險後，接受低風險用途。

控制措施實施後，還要評估殘餘風險是否可接受。若風險仍超過組織容忍範圍，就不應擴大導入。

### 7.14 治理、人工監督與責任分工

本章以組織層的人工監督（Human Oversight）為主責。Human Oversight 是持續監督與治理整個 AI 系統的上位概念，包含角色責任、權限、監控、事件處理與定期改善；它不等於每一筆輸出都採 HITL。生成式 AI 治理應涵蓋：

1. 核准的工具與使用範圍。
2. 可輸入資料與禁止輸入資料。
3. 角色、權限與重要動作核准。
4. 輸出查核、來源保存與對外發布規則。
5. 提示、版本、資料、動作與審核紀錄。
6. 事件通報、停止服務與修復流程。
7. 教育訓練與使用者支援。
8. 定期成效、風險與供應商檢查。

PoC 階段可以採較小範圍的治理，但不能沒有治理。例如使用測試資料、限制使用者、關閉高風險動作、設人工覆核並保存測試結果。正式導入後則要加入組織級政策、持續監控、稽核與生命週期管理。

人工覆核用於檢查個別輸出或高風險結果；人工批准則要求具權限人員在關鍵行動執行前明確確認。當覆核或批准被設為決策流程中的必要節點，未經人員處理便不能繼續時，才稱為人在迴圈中（Human-in-the-loop, HITL）。審核者必須有足夠資訊、能力、時間與拒絕權限，而不是只做形式上的點選確認。若人類持續監看自主運作的系統並可在必要時介入，則屬第一章所述的 HOTL。

### 7.15 內容出處與生成內容驗證

內容出處（Content Provenance）是記錄數位內容來源與修改歷程的資訊。浮水印、雜湊、簽章或內容憑證可協助辨識來源與修改紀錄，但各工具支援方式會變動。

C2PA（Coalition for Content Provenance and Authenticity）建立的 Content Credentials 是一種內容來源與歷程標準。使用時要理解兩個界線：

- 出處資訊可協助驗證內容的來源紀錄是否與資產相連、是否被竄改。
- 出處資訊本身不能證明畫面中的事件、敘述或主張一定真實正確。

因此，內容標示與來源技術是查核線索，不是取代事實查證的萬能工具。

### 7.16 從試點到規模化導入

PoC 或試點達標後，仍應在擴大前完成：

1. 將成功條件、限制與失敗案例寫入文件。
2. 建立標準作業流程、提示範本與資料更新方式。
3. 確認服務容量、故障處理與支援窗口。
4. 完成使用者訓練與角色權限配置。
5. 設定品質、成本、風險與採用率監控。
6. 建立變更、版本、事件與退場管理。
7. 依實際結果決定擴大、調整或停止。

規模化不只是增加帳號數量，而是把經驗轉成可重複、可治理與可維護的組織能力。

### 本章重點整理

- 生成式 AI 導入應先定義問題、價值與成功條件，不應從採購工具開始。
- 評估需同時考量問題、價值、資料、技術、成本、風險與組織接受度。
- 高價值、低難度且低風險的場景適合優先 PoC；高風險場景需更嚴格控制。
- TCO 包含工具、資料、整合、訓練、維運、資安及風險等全生命週期成本。
- ROI 與回收期不同；Token 使用成本也只是 TCO 的一部分。
- 雲端、私有、邊緣與混合部署各有取捨，沒有適用所有情境的單一答案。
- PoC 驗證構想，MVP 驗證核心使用，試點驗證營運，正式導入進入持續治理。
- KPI 應同時衡量效率、品質、成本、採用與風險。
- Latency 關注單次回應時間，Throughput 關注單位時間處理量。
- 風險控制必須涵蓋資料、模型、應用、工具權限、人員與供應鏈。
- PoC 階段就需要最低限度治理，正式導入後再擴大制度與持續監控。
- Human Oversight 是組織層的持續監督；人工覆核檢查個別結果，人工批准把關關鍵行動，設為必要決策節點時才屬 HITL。
- 內容出處可提供來源線索，但不能單獨證明內容為真。

### 學習重點

1. 導入順序從業務問題與 KPI 開始，再評估資料、技術與工具。
2. 資料存在不等於合法、正確或適合使用，必須確認來源、品質、權限與更新。
3. 場景排序要同時看價值、難度與風險。
4. TCO 不只包含 API 或訂閱費；ROI 與回收期是不同指標。
5. PoC 不等於正式上線，成功後仍需經過使用、整合、維運與治理驗證。
6. 私有化提高控制程度，但不會自動消除幻覺、公平性偏誤或安全設定錯誤。
7. 卡頓主要看 Latency，大量處理能力看 Throughput；批次化可能在兩者間取捨。
8. Data Drift 是部署後環境改變，過度擬合則是訓練階段的泛化問題。
9. Agent 能執行的動作越多，越需要最小權限、動作限制、人工確認與稽核。
10. 出處資訊只能協助理解來源與歷程，不能單獨證明內容主張為真。

### 自我測驗

**1. 某企業表示「想全面導入生成式 AI 提升競爭力」，但尚未定義流程、痛點與衡量方式。最適當的第一步為何？**

A. 立即購買最多功能的工具  
B. 先盤點需求、現況流程與可衡量 KPI  
C. 直接把所有內部資料上傳  
D. 先全面開放 Agent 執行權限

**2. 下列哪一項最接近完整的 TCO 評估？**

A. 只計算每月 API 費用  
B. 只計算初期硬體採購  
C. 同時計入工具、資料、整合、訓練、維運、資安與風險成本  
D. 只計算員工節省的時間

**3. 關於 PoC 與正式導入，下列敘述何者正確？**

A. PoC 成功就可以跳過試點與治理直接全面上線  
B. PoC 只測模型速度，不需測業務價值  
C. PoC 用小範圍驗證假設，正式導入還需處理營運、治理與持續監控  
D. PoC 階段完全不需要資料與權限控制

**4. 即時客服使用者抱怨送出問題後等待很久，但系統每分鐘總處理量足夠。應優先檢查哪項指標？**

A. Latency  
B. Throughput  
C. ROI  
D. 資料標註率

**5. 某 Agent 可讀取客戶資料、寄送郵件並取消訂單，但所有動作都使用同一個最高權限帳號。最主要的風險與改善為何？**

A. Temperature 太低；應調高隨機性  
B. 過度代理；應採最小權限、限制動作並對重要操作設人工確認  
C. 欠擬合；應增加訓練輪數  
D. Chunking 太小；應增加文件長度

**6. 推薦系統上線數月後，使用者行為改變，模型效果逐步下降。這最可能是什麼問題？**

A. Data Drift  
B. Few-shot Prompting  
C. 服務虛擬化  
D. One-hot Encoding

**7. 下列哪一項最符合有效 KPI 的設計？**

A. 只寫「讓 AI 更好用」  
B. 只要求每位員工每天使用十次，不衡量品質  
C. 設定基準值、目標、期間、資料來源，並同時觀察品質與風險  
D. 專案完成後才決定如何衡量

**8. 關於 C2PA 或其他內容出處資訊，下列敘述何者正確？**

A. 有出處資訊就能保證內容中的事件真實發生  
B. 沒有出處資訊就一定是 AI 生成  
C. 出處資訊可協助理解來源與修改歷程，但仍需查核內容真實性  
D. 出處資訊可以取代所有人工審核

#### 答案與解析

1. **B**。需求尚不明確時，應先盤點痛點、流程、使用者與 KPI；直接採購或全面開放權限無法證明導入價值。
2. **C**。TCO 是全生命週期成本，除了工具或硬體，也包括資料、整合、訓練、維運、安全、合規與風險等投入。
3. **C**。PoC 用受控範圍驗證技術、價值與風險；正式導入還需要營運流程、權限、支援、治理與持續監控。
4. **A**。使用者感受到的是單次互動等待時間，應優先檢查 Latency；Throughput 則衡量單位時間總處理量。
5. **B**。Agent 具有過大的資料與動作權限，屬過度代理風險。應採最小權限、動作白名單、參數驗證、人工確認與稽核。
6. **A**。部署後輸入或使用者行為分布改變，使模型表現下降，屬於 Data Drift。
7. **C**。可用的 KPI 應有基準、目標、期間、資料來源與負責人，且不能只追求使用量而忽略品質與風險。
8. **C**。內容出處資訊能提供來源及修改歷程線索，但不能單獨證明畫面或敘述為真，仍需事實查核。
