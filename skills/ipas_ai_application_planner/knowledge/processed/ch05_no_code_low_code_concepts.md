# 第五章：No-Code／Low-Code 概念

## 學習目標

完成本章後，學習者應能：

1. 說明無程式碼（No-Code）與低程式碼（Low-Code）的基本概念。
2. 比較 No-Code、Low-Code 與傳統程式開發的適用情境與限制。
3. 認識視覺化介面、資料模型、工作流程與系統整合等常見平台組成。
4. 區分 Low-Code 情境中的「Model」與機器學習模型。
5. 說明 No-Code、Low-Code、自動化機器學習與預訓練 AI API 的分工。
6. 依據需求複雜度、客製彈性與治理要求選擇合適方式。
7. 理解公民開發者、影子 IT，以及權限、資料與生命週期治理的重要性。
8. 說明整合測試與服務虛擬化在平台應用中的用途。

## 核心概念

No-Code 與 Low-Code 的共同目的，是透過視覺化工具、可重複使用元件與預先建立的連接機制，降低應用程式開發門檻並加速需求落地。兩者並不是完全排除軟體工程，而是改變開發者撰寫、組合與管理系統的方式。

- **No-Code**：以拖拉、表單、規則設定等方式建立應用，通常不要求使用者撰寫程式碼。
- **Low-Code**：同樣以視覺化開發為主，但允許加入少量程式碼，以處理客製邏輯、特殊介面或複雜整合。
- **資料模型（Data Model）**：描述資料欄位、型別、關聯與規則，是應用系統的資料基礎。
- **工作流程（Workflow）**：定義事件發生後的處理步驟、條件判斷、通知與核准路徑。
- **系統整合（System Integration）**：透過連接器或應用程式介面（Application Programming Interface, API）交換資料與呼叫外部服務。
- **治理（Governance）**：以權限、資料標準、審查、稽核及生命週期管理，確保快速開發不會造成安全與管理風險。

可以把基本開發流程整理為：

> 釐清需求與使用者 → 選擇開發方式 → 定義資料與畫面 → 建立流程與整合 → 設定權限 → 測試 → 發布與持續管理

## 詳細內容

### 5.1 No-Code 與 Low-Code 的基本概念

傳統軟體開發通常需要開發者自行撰寫程式、設計資料存取方式，並處理介面、流程與系統整合。No-Code 與 Low-Code 平台則將許多常見功能做成視覺化元件，讓使用者透過設定與組合完成應用。

這類平台適合處理表單、審核、資料查詢、通知、簡易入口網站與部門流程等需求。不過，開發速度快不代表可以省略需求分析、測試、安全控管或後續維運。

#### 5.1.1 No-Code

No-Code 主要服務不具程式背景、但熟悉業務流程的使用者。常見操作包括：

- 拖拉畫面元件。
- 設定表單欄位與資料來源。
- 以條件規則建立流程。
- 使用既有連接器傳送通知或交換資料。
- 透過範本快速建立常見應用。

例如，人資部門要建立一個內部教育訓練申請流程，需求只有填表、主管核准、寄送通知與結果查詢，便可能使用 No-Code 完成原型或正式應用。

No-Code 的優點是上手較快、開發週期短；限制則是客製能力通常受平台元件與規則範圍影響。當需求需要特殊演算法、複雜交易控制或大量客製整合時，可能無法只靠 No-Code 完成。

#### 5.1.2 Low-Code

Low-Code 也提供視覺化工具，但保留程式擴充能力。開發者可在平台既有功能不足時，撰寫少量程式來處理：

- 特殊商業規則。
- 自訂畫面或元件。
- 複雜資料轉換。
- 非標準 API 串接。
- 額外的安全或效能需求。

因此，Low-Code 常由具備技術能力的開發者與業務人員共同使用。它在開發效率與客製彈性之間取得平衡，但團隊仍需具備程式設計、測試、資料庫與系統整合等基本能力。

### 5.2 No-Code、Low-Code 與傳統開發比較

三種方式不是互相取代，而是適用於不同複雜度與治理要求。

| 比較面向 | No-Code | Low-Code | 傳統程式開發 |
| --- | --- | --- | --- |
| 主要使用者 | 業務人員、公民開發者 | 開發者與業務協作者 | 專業軟體開發團隊 |
| 建置方式 | 視覺化設定為主 | 視覺化設定加少量程式碼 | 以程式碼與工程工具為主 |
| 開發速度 | 通常較快 | 通常較快 | 視需求與工程規模而定 |
| 客製彈性 | 較受平台限制 | 中至高 | 通常最高 |
| 適合情境 | 標準化表單與簡易流程 | 需客製邏輯或整合的應用 | 高度複雜、特殊效能或底層控制需求 |
| 技術要求 | 較低 | 中等 | 較高 |

選擇時不應只比較「能否快速完成」，還要評估：

1. 流程與商業規則是否複雜。
2. 是否需要特殊介面或大量客製功能。
3. 是否必須串接舊系統或多個外部服務。
4. 資料是否敏感，權限與稽核要求是否嚴格。
5. 使用人數、交易量與效能需求。
6. 未來是否容易移轉、擴充及維護。

### 5.3 平台的常見組成

不同平台的名稱與功能不完全相同，但通常可從資料、介面、流程與整合四個面向理解。

#### 5.3.1 資料層

資料層負責定義應用需要保存的內容，例如客戶、訂單、申請單與審核紀錄。建立時要考慮：

- 欄位名稱與資料型別。
- 必填、預設值及格式驗證。
- 資料表之間的關聯。
- 誰可以建立、讀取、修改或刪除資料。
- 資料保存、備份與刪除規則。

視覺化建模雖然降低操作難度，但若資料結構設計不佳，仍可能出現重複、矛盾或難以整合的問題。

#### 5.3.2 使用者介面

使用者介面通常由表單、清單、按鈕、圖表及導覽元件組成。設計時除了畫面是否美觀，也要確認操作流程、欄位提示、錯誤訊息與不同裝置上的可用性。

#### 5.3.3 工作流程與自動化

工作流程描述資料在不同狀態間如何流動。例如採購申請可能依序經過：

> 申請人送出 → 主管審核 → 金額條件判斷 → 採購單位處理 → 結果通知

流程通常包含觸發事件、條件判斷、動作、等待、核准及例外處理。建立流程時，必須同時思考失敗、逾時、重複執行與人工介入方式。

#### 5.3.4 連接器與 API

連接器（Connector）是平台預先封裝的整合元件；API 則提供系統之間交換資料或呼叫功能的介面。若應用要讀取企業資料庫、寄送通知、查詢物流狀態或使用 AI 服務，通常需要系統整合。

串接時應確認：

- 身分驗證與授權方式。
- 傳輸資料的格式與欄位對應。
- 呼叫失敗、逾時及重試機制。
- API 版本變更的影響。
- 敏感資料是否被不必要地傳送或記錄。

### 5.4 Low-Code 中的 Model 不是機器學習模型

「模型」一詞在不同技術領域有不同含義，這是常見的混淆來源。

在機器學習中，模型（Machine Learning Model）是從資料學得規律、用來預測或生成結果的數學與計算結構。Low-Code 應用開發所談的 Model，則常指資料結構、領域狀態與商業規則。

以教材介紹的模型—視圖—控制器（Model-View-Controller, MVC）概念為例：

| 元件 | 主要角色 | 訂單系統例子 |
| --- | --- | --- |
| Model | 管理資料、狀態與相關商業規則 | 訂單、商品、金額與訂單狀態 |
| View | 呈現資訊並接收使用者操作 | 訂單表單與查詢畫面 |
| Controller | 接收操作，協調流程與 Model、View | 處理送出訂單或取消訂單的請求 |

因此，在考題或需求文件看到「建立 Model」時，應先從上下文判斷它是應用程式架構中的資料模型，還是 AI／機器學習模型，不能只依中文名稱判斷。

### 5.5 No-Code／Low-Code 與 AI 工具的分工

No-Code、Low-Code、自動化機器學習（Automated Machine Learning, AutoML）與預訓練 AI API 解決的問題不同。

| 工具類型 | 主要用途 | 典型產出 |
| --- | --- | --- |
| No-Code | 以設定方式建立表單、流程與簡易應用 | 業務流程應用 |
| Low-Code | 快速建立可客製、可整合的應用 | 企業應用與系統介面 |
| AutoML | 自動化資料建模、演算法比較或參數調整等工作 | 經訓練的機器學習模型 |
| 預訓練 AI API | 直接提供文字、影像、語音等既有 AI 能力 | AI 推論結果 |

Low-Code 平台本身不等於 AutoML，也不代表內建的每項功能都是 AI。實務上可將不同工具組合使用。例如：

1. 以 Low-Code 建立客服案件管理畫面。
2. 呼叫預訓練語言 AI API，協助摘要案件內容。
3. 將摘要與分類結果寫回案件資料。
4. 透過工作流程分派給適當人員。

如果企業擁有足夠的歷史資料，也可能以 AutoML 訓練分類模型，再由 Low-Code 應用呼叫模型服務。關鍵是先釐清需求，再決定要「建立應用」、「訓練模型」，還是「使用既有 AI 能力」。

### 5.6 適用情境與選擇原則

#### 情境一：標準化行政流程

需求包含填寫表單、主管核准與寄送通知，規則固定且整合簡單。此時可優先評估 No-Code。

#### 情境二：需要客製邏輯與多系統串接

需求包含特殊計價規則、舊系統資料轉換與外部 API，視覺化元件無法完整處理。此時較適合 Low-Code，並由技術人員補充程式與測試。

#### 情境三：核心交易或高度特殊系統

若系統具有複雜交易、高效能、特殊資安控制或底層技術需求，應評估傳統開發或混合架構，而不是勉強將所有功能放入 No-Code／Low-Code 平台。

#### 情境四：在應用中加入 AI

若只需成熟的文字辨識、摘要或影像分析能力，可評估呼叫預訓練 AI API；若需要依企業資料訓練預測模型，可評估 AutoML 或其他機器學習流程。No-Code／Low-Code 則負責畫面、資料、流程與服務串接。

### 5.7 公民開發者與影子 IT

公民開發者（Citizen Developer）是熟悉業務需求、使用組織核准工具建立應用的非專職開發人員。他們能縮短需求溝通距離，也能協助部門快速改善流程。

如果缺乏管理，各部門可能自行建立未被資訊單位掌握的系統，形成影子 IT（Shadow IT）。常見風險包括：

- 相同需求被重複建置。
- 欄位定義與資料標準不一致。
- 敏感資料存放位置不明。
- 權限過大，離職或轉調後未回收。
- 應用沒有負責人、文件或維護計畫。
- 外部連接器或帳號未受到審查。

因此，鼓勵公民開發並不等於放棄資訊治理，而是要建立清楚的授權範圍與支援機制。

### 5.8 治理、安全與生命週期管理

組織可以從下列面向建立 No-Code／Low-Code 治理：

1. **平台與應用目錄**：記錄平台、應用名稱、用途、負責人與使用資料。
2. **分級與審查**：依資料敏感度、使用範圍及整合風險決定審查程度。
3. **權限控管**：採最小權限原則，區分開發、測試、發布與管理角色。
4. **資料標準**：統一重要欄位定義、格式、主檔與資料品質規則。
5. **連接器管理**：審核外部服務、帳號、金鑰及資料傳輸範圍。
6. **版本與變更管理**：保留版本、測試結果、發布紀錄及必要的回復方式。
7. **稽核與監控**：記錄存取、異動、錯誤與異常使用情形。
8. **退場機制**：當應用停用或被取代時，妥善移轉、保存或刪除資料。

即使平台已提供權限設定，應用建立者仍要正確配置。安全不是平台名稱自動保證的結果，而是平台能力、組織制度與實際設定共同作用的成果。

### 5.9 測試與服務虛擬化

視覺化預覽只能確認部分畫面與流程，不足以取代完整測試。常見測試層次包括：

- **邏輯測試**：確認欄位驗證、公式與條件分支是否正確。
- **整合測試（Integration Testing）**：確認應用與資料庫、API 或其他系統的資料交換。
- **端對端測試（End-to-End Testing）**：從使用者操作開始，檢查完整流程與最終結果。
- **使用者驗收測試（User Acceptance Testing, UAT）**：由實際使用者確認系統是否符合業務需求。

當外部系統尚未完成、測試環境不穩定，或不容易製造逾時與錯誤情境時，可使用服務虛擬化（Service Virtualization）。它以模擬服務回傳預期結果，例如：

- 正常回應與正確資料。
- 查無資料。
- 欄位格式錯誤。
- 服務逾時或無法連線。
- 權限驗證失敗。

服務虛擬化可讓團隊提早測試整合邏輯與例外處理，但上線前仍應在可行範圍內與真實系統完成整合驗證。

### 5.10 從快速原型走向正式應用

No-Code／Low-Code 很適合快速驗證需求，但原型要成為正式應用，仍需補足工程與管理工作：

1. 確認需求、資料定義與流程責任。
2. 檢查權限、敏感資料及外部整合。
3. 建立測試案例，包含正常與異常路徑。
4. 規劃發布、監控、備份與故障處理。
5. 指定應用負責人與維護方式。
6. 定期檢查平台版本、連接器及帳號權限。

真正的價值不只是「做得快」，而是以適合的工具持續、安全地解決業務問題。

### 本章重點整理

- No-Code 以視覺化設定為主，適合標準化、複雜度較低的應用。
- Low-Code 保留程式擴充能力，適合需要客製邏輯或系統整合的情境。
- No-Code、Low-Code 與傳統開發各有適用範圍，應依需求與風險選擇。
- 平台常見組成包括資料模型、使用者介面、工作流程及連接器／API。
- Low-Code 架構中的 Model 多半是資料與商業領域模型，不是機器學習模型。
- AutoML 用於協助訓練模型；預訓練 AI API 提供可直接呼叫的 AI 能力。
- 公民開發能加速數位化，但缺乏治理時可能形成影子 IT。
- 正式應用仍需要權限、資料、版本、稽核、測試及退場管理。
- 服務虛擬化可模擬外部服務，協助提早測試整合與異常情境。

### 關鍵名詞

| 名詞 | 簡短說明 |
| --- | --- |
| No-Code | 不要求撰寫程式碼、以視覺化設定建立應用的方式。 |
| Low-Code | 以視覺化開發為主，並允許用少量程式碼擴充的方式。 |
| 公民開發者（Citizen Developer） | 使用組織核准工具建立應用的非專職開發人員。 |
| 影子 IT（Shadow IT） | 未被組織正式掌握或治理的資訊系統、服務與技術使用。 |
| 資料模型（Data Model） | 描述資料欄位、型別、關聯與規則的結構。 |
| MVC | 以 Model、View、Controller 分工組織應用程式的架構概念。 |
| 工作流程（Workflow） | 定義事件、條件、處理步驟與核准路徑的流程。 |
| 連接器（Connector） | 平台預先封裝、用來連接資料來源或外部服務的元件。 |
| API | 讓不同軟體交換資料或呼叫功能的介面。 |
| AutoML | 將部分機器學習建模工作自動化的方法或工具。 |
| 治理（Governance） | 對平台、資料、權限、變更與生命週期建立管理規範。 |
| 服務虛擬化（Service Virtualization） | 模擬外部服務行為，以支援開發與整合測試的方法。 |

### 學習重點

1. No-Code 與 Low-Code 的差異，核心在於使用者、客製能力與是否需要程式擴充。
2. 平台降低的是實作門檻，不會消除需求分析、測試、資安與維運責任。
3. 看到「Model」時要依上下文判斷，避免與機器學習模型混淆。
4. 要建立應用介面與流程，選 No-Code／Low-Code；要訓練預測模型，評估 AutoML；要直接使用既有 AI 能力，可串接預訓練 AI API。
5. 工具選擇應同時考慮複雜度、整合、資料敏感度、效能、擴充與維護。
6. 公民開發必須搭配應用目錄、分級審查、最小權限、資料標準與稽核機制。
7. 外部服務的成功、失敗與逾時情境都應納入整合測試。
8. 原型能快速驗證需求，但正式上線前仍要完成治理與生命週期規劃。

### 自我測驗

**1. 某部門要建立內部訪客申請表，流程只有填表、主管核准與寄送通知，且沒有特殊整合需求。最適合優先評估哪一種方式？**

A. No-Code
B. Low-Code，因為所有表單流程都必須加入自訂程式
C. 傳統開發，因為標準化流程不適合視覺化元件
D. AutoML，因為主管核准屬於模型訓練任務

**2. 某應用可用拖拉方式完成大部分畫面，但仍需撰寫少量程式串接舊系統並實作特殊計價規則。較適合哪一種方式？**

A. No-Code，因為非標準整合不需要程式擴充
B. Low-Code，因為可保留視覺化開發並加入客製程式
C. 傳統開發，因為只要有一段程式就不能使用 Low-Code
D. AutoML，因為特殊計價規則需要比較演算法

**3. 在教材所述 MVC 應用架構中，Model 最主要代表什麼？**

A. 使用者看見的表單、清單與操作畫面
B. 接收操作並協調 Model 與 View 的流程元件
C. 應用的資料、狀態與相關商業規則
D. 從標註資料學得分類邊界的機器學習模型

**4. 某核心交易系統需要極低延遲、複雜交易一致性、特殊資安控制與底層效能調校。最適合的初步判斷為何？**

A. 優先用 No-Code，因為視覺化開發能涵蓋所有底層控制
B. 優先用 AutoML，因為交易一致性是模型選擇問題
C. 優先用服務虛擬化作正式系統，取代真實交易服務
D. 評估傳統開發或混合架構，不應勉強全部放入平台

**5. 團隊已用 Low-Code 建立客服案件流程，現在希望直接加入文字摘要能力，且暫時不自行訓練模型。較合理的作法為何？**

A. 呼叫合適的預訓練 AI API，並處理驗證、逾時與權限
B. 使用 AutoML 建立表單畫面，再由 Low-Code 訓練語言模型
C. 將 Low-Code 的資料 Model 直接當成摘要模型
D. 只建立新的工作流程，不串接任何 AI 服務

**6. 三個部門各自建立相似的客戶應用，欄位名稱不一致、負責人不明，資訊單位也不知道資料存放位置。這最接近哪一項問題？**

A. 交叉驗證
B. 影子 IT 與治理不足
C. 模型欠擬合
D. 資料增強

**7. 物流公司的外部 API 尚未提供穩定測試環境，但團隊需要先驗證正常回應、逾時與權限失敗時的處理。最合適的作法為何？**

A. 只做 UAT，因為使用者驗收可以取代 API 錯誤測試
B. 等正式上線後再利用真實錯誤測試例外處理
C. 使用服務虛擬化模擬回應，之後仍與真實系統完成整合驗證
D. 以單元測試取代所有跨系統整合測試

**8. 組織擔心平台供應商鎖定與應用無人維護。下列哪項做法最完整？**

A. 只比較初期建置速度，移轉與退場待停用時再處理
B. 允許各部門自行選連接器，以降低中央治理造成的鎖定
C. 將所有帳號設為管理者，避免負責人離職後無法維護
D. 建立應用目錄、負責人、版本與退出計畫，並評估資料及流程可移轉性

#### 答案與解析

1. **A**。標準化表單、固定核准與通知且無特殊整合，適合先評估 No-Code；Low-Code 留給需要客製程式或複雜整合的情境。
2. **B**。Low-Code 能保留視覺化開發效率，並以少量程式處理特殊規則與舊系統串接；有程式擴充不代表必須全面改採傳統開發。
3. **C**。MVC 的 Model 管理應用資料、狀態與商業規則；View 是畫面，Controller 協調操作，三者都不同於機器學習模型。
4. **D**。複雜交易、特殊效能與底層控制可能超出平台適用範圍，應評估傳統或混合架構；服務虛擬化是測試手段，不是正式交易服務。
5. **A**。既有應用要取得摘要能力，可串接預訓練 AI API，並處理驗證、權限與失敗流程；Low-Code 的資料 Model 不具備語言生成能力。
6. **B**。未納管的重複應用、標準不一致與責任不明，是影子 IT 與治理不足；交叉驗證與資料增強屬機器學習概念。
7. **C**。服務虛擬化可提早模擬正常與異常回應，但不能永久取代真實系統整合驗證；UAT 也不能覆蓋所有 API 失敗路徑。
8. **D**。應用目錄、明確負責人、版本、退出計畫與可移轉性可同時降低維護與鎖定風險；全面管理者權限反而提高安全風險。
