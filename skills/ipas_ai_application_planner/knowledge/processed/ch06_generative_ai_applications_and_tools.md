# 第六章：生成式 AI 應用領域與工具使用

## 學習目標

完成本章後，學習者應能：

1. 說明生成式人工智慧（Generative Artificial Intelligence, Generative AI）的主要應用領域。
2. 區分文字、圖像、語音、音樂、影片、程式碼與多模態生成任務。
3. 認識提示詞（Prompt）的基本組成與常見提示策略。
4. 依輸入、輸出、任務、資料與風險選擇合適的生成式 AI 工具類型。
5. 說明檢索增強生成、工具呼叫與 AI 代理的功能差異。
6. 建立從需求定義、內容生成到人工審核的實際應用流程。
7. 使用正確性、完整性、一致性、適用性、合規性與可執行性檢查生成內容。
8. 辨識幻覺、偏誤、著作權、隱私、資訊安全及過度依賴等風險。
9. 說明人工審核、權限控制與責任邊界的重要性。

## 核心概念

生成式 AI 能依據文字、圖片、聲音、資料或其他輸入，產生新的文字、影像、音訊、影片、程式碼或跨模態內容。它的價值不只在「生成」，更在於協助整理、改寫、轉換、構思與建立初稿。

生成式 AI 的應用可以整理為一條人機協作流程：

> 定義任務 → 盤點資料與風險 → 選擇工具類型 → 設計提示與上下文 → 產生初稿 → 查證與測試 → 人工核准 → 發布並保留紀錄

本章有三個重要判斷原則：

- **先看任務，再選工具**：不要因某項工具熱門，就把所有問題都交給同一種工具。
- **生成結果是候選內容，不是當然正確的答案**：重要事實、數字、來源與專業判斷仍要查核。
- **自動化程度越高，控制要求越高**：當 AI 能讀取內部資料或執行外部動作時，必須加強權限、驗證、稽核與人工確認。

> **教材脈絡：**原始教材以應用領域、提示策略、RAG、工具整合、生成參數及多模態應用為主軸。
>
> **補充說明：**本章以工具「類型」取代容易變動的產品名稱與版本，並將品質檢查及責任邊界整合成可實際操作的流程。

## 詳細內容

### 6.1 生成式 AI 的主要應用領域

生成式 AI 的分類方式很多。從輸出形式來看，可分為文字、圖像、音訊、影片與程式碼；從工作情境來看，則可應用於行政、行銷、教育、客服、製造、軟體開發、法務與知識管理等領域。

| 應用領域 | 常見任務 | 典型產出 | 主要檢查重點 |
| --- | --- | --- | --- |
| 行政與文書 | 摘要、會議紀錄、信件草稿 | 文件、表格、待辦清單 | 機密資料、事實與行動項目 |
| 行銷與設計 | 文案、活動概念、視覺草稿 | 貼文、海報、分鏡 | 品牌一致性、授權與不實宣稱 |
| 教育訓練 | 教材、測驗、學習活動 | 講義、題庫、教案 | 知識正確性與程度適配 |
| 客服與知識管理 | 回覆草稿、FAQ、文件問答 | 回覆、摘要、案件分類 | 最新政策、來源與責任歸屬 |
| 製造與品質 | SOP 草稿、異常摘要、維修知識 | 流程文件、改善建議 | 現場可行性與工程師確認 |
| 軟體開發 | 程式生成、除錯、測試、文件 | 程式碼、測試案例、說明文件 | 正確性、安全性、授權與相依性 |
| 法務與合規 | 條款摘要、差異整理、風險清單 | 摘要與檢查表 | 來源、時效與專業人員覆核 |

同一個業務問題可能同時使用多種能力。例如，會議助理可能先把語音轉成文字，再辨識發言者、摘要主題、擷取待辦事項，最後產生文字報告。這就是跨模態與多階段流程的組合，而不是單一生成動作。

### 6.2 文字生成

文字生成是最常見的生成式 AI 應用，包括：

- 撰寫、改寫與調整語氣。
- 長文件摘要與重點擷取。
- 翻譯與格式轉換。
- 問答、企劃與腦力激盪。
- 依固定欄位產生報告草稿。
- 從非結構化文字抽取結構化資訊。

文字生成適合產生初稿或協助整理，但不應把語句流暢視為內容正確。若任務涉及最新政策、精確數字或專業判斷，應要求保留來源，並回到原始資料或官方資訊查證。

**情境例子：**客服人員請 AI 依公司政策產生回覆草稿。若只使用一般聊天模型，模型可能依既有知識推測；若政策經常更新，則較適合先從核准的知識庫檢索，再由 AI 生成回答。

### 6.3 圖像生成

圖像生成可依文字描述產生插圖、概念圖、版面草稿、商品情境圖或設計變體，也可以依既有圖片進行局部修改與風格調整。

建立圖像需求時，通常要交代：

1. 主體與場景。
2. 構圖、視角與比例。
3. 色彩、風格與氣氛。
4. 用途與輸出尺寸。
5. 不應出現的元素。

圖像品質不只看「好不好看」，也要檢查文字是否錯誤、人物或物件是否不合理、品牌元素是否符合規範，以及素材、人物肖像與輸出用途是否具備適當權利或授權。

### 6.4 語音與音樂生成

語音相關工具可用於文字轉語音、配音、語音風格調整、語音助理與逐字稿處理；音樂生成則可協助產生旋律、配樂或聲音概念草稿。

使用時應注意：

- 合成語音是否造成真人冒充或誤認。
- 是否取得聲音、作品與相關素材的適當同意或授權。
- 發音、專有名詞、語氣與情緒是否符合情境。
- 對外內容是否需要揭露或標示其生成方式。
- 高風險語音指令是否有身分確認與人工介入。

語音或音樂輸出即使聽起來自然，也不能因此推定來源合法、內容真實或適合商業使用。

### 6.5 影片生成

影片生成可協助建立腳本、分鏡、場景草稿、短片或視覺特效。常見流程是先產生文字腳本，再轉換為畫面、旁白與字幕，最後由人員剪輯與校正。

影片包含較多時間與空間關係，因此要特別檢查：

- 前後鏡頭中的人物、物件與場景是否一致。
- 動作、物理關係與口型是否合理。
- 旁白、字幕與畫面是否同步。
- 是否可能被誤認為真實事件或真實人物發言。
- 發布前是否完成肖像、素材、品牌與內容風險檢查。

內容出處資訊、浮水印或其他標示可協助說明內容來源，但不能單獨證明影片所描述的事件為真，仍要進行事實查核。

### 6.6 程式碼生成

生成式 AI 可協助產生函式草稿、測試案例、資料查詢語句、程式說明、重構建議與 API 文件。它能提升開發效率，但產出的程式碼仍可能有邏輯錯誤、安全漏洞、不適用的相依套件或授權疑慮。

程式碼生成的基本審查流程包括：

1. 先確認需求與輸入輸出規格。
2. 將任務拆成可測試的小單位。
3. 檢查程式邏輯、錯誤處理及邊界條件。
4. 執行單元測試、整合測試與安全檢查。
5. 由具備責任與能力的人員完成程式碼審查。

不得因程式可以執行，就直接判定它安全、正確或可部署到正式環境。

### 6.7 多模態 AI

多模態人工智慧（Multimodal AI）能處理或產生兩種以上的資料形式，例如文字、圖像、音訊與影片。常見任務包括：

- 依圖片產生描述或替代文字。
- 分析表單影像並轉為結構化資料。
- 將會議錄音轉成文字後產生摘要。
- 根據文字腳本生成分鏡與旁白。
- 結合圖片與文字回答商品或設備問題。

多模態並不是把多個檔案放在一起就一定有效。工具必須能理解相應模態，輸入資料也要清楚、完整並具備正確對應關係。例如圖像描述需要同時處理影像特徵與語言生成，不能只調整文字提示就解決所有視覺理解問題。

### 6.8 Prompt 的基本概念

提示詞（Prompt）是使用者提供給生成式 AI 的任務指令、背景資料與輸出要求。良好的提示詞應讓模型知道「要做什麼、依據什麼、給誰使用、輸出長什麼樣、有哪些限制」。

| 提示元素 | 作用 | 簡短例子 |
| --- | --- | --- |
| 角色（Role） | 指定回答角度 | 你是一位客服流程顧問 |
| 任務（Task） | 明確說明工作 | 請整理客訴原因與建議行動 |
| 背景（Context） | 提供情境與對象 | 對象是新進客服人員 |
| 資料（Data） | 限定可使用的內容 | 僅依下列訪談紀錄整理 |
| 格式（Format） | 指定輸出結構 | 請輸出為三欄表格 |
| 限制（Constraints） | 說明不可做或邊界 | 找不到依據時請標示無法判斷 |
| 檢核（Check） | 指定品質檢查 | 最後列出需人工查證的項目 |

提示詞是任務規格，不是保證正確的魔法指令。若資料不足、來源錯誤或任務不適合，提示寫得再詳細也不能消除所有問題。

### 6.9 常見提示策略

| 策略 | 英文 | 適合情境 | 使用重點 |
| --- | --- | --- | --- |
| 零樣本提示 | Zero-shot Prompting | 模型熟悉的簡單通用任務 | 直接給清楚任務與格式 |
| 少樣本提示 | Few-shot Prompting | 需要固定格式或風格 | 提供少量品質良好的輸入輸出範例 |
| 任務拆解 | Task Decomposition | 複雜或多階段工作 | 拆成可檢查的小步驟 |
| 多方案比較 | Tree of Thoughts, ToT | 規劃、設計與方案評估 | 先列候選方案，再依共同準則比較 |
| 關係結構提示 | Graph Prompting | 多因素交互或網絡關係 | 明確描述節點、關係與限制 |
| 提示鏈 | Prompt Chaining | 長流程或多種輸出 | 將前一步可驗證結果交給下一步 |
| 結構化輸出 | Structured Output | 結果要交給系統處理 | 指定 JSON、表格或固定欄位 |

教材以思維鏈（Chain-of-Thought, CoT）描述逐步處理問題的提示方式。實務上更重要的是要求模型提供可檢查的步驟、計算、依據或中間結果；不能把模型未公開的內部推理當成正確性的證明。

### 6.10 上下文工程與生成參數

提示工程（Prompt Engineering）著重單次指令如何表達；上下文工程（Context Engineering）則進一步安排模型執行任務所需的整體輸入環境，例如：

- 系統規則與任務提示。
- 文件檢索結果。
- 對話摘要與必要記憶。
- 可使用的工具及權限。
- 結構化資料與輸出格式。

上下文工程主要改變輸入與執行環境，不等同於修改模型參數。面對表格或欄位資料時，可先轉換成清楚的 Markdown 表格或 JSON 結構，並說明欄位意義，通常比直接貼上混亂文字更容易處理。

溫度（Temperature）是常見生成參數，用來調整輸出的隨機程度。較低設定通常有利於一致與保守的輸出；較高設定可能增加多樣性。它不能修正錯誤知識，也不能取代來源查核。實際數值範圍與效果依工具而異，不宜把固定數字視為所有產品的通用答案。

### 6.11 RAG、工具呼叫、MCP 與 AI Agent

生成式 AI 應用不一定只靠模型既有知識。依需求不同，可加入檢索、工具呼叫或代理流程。

| 方法 | 主要目的 | 典型情境 | 主要風險 |
| --- | --- | --- | --- |
| RAG | 補充可更新、可追溯的外部知識 | 企業文件問答、SOP 查詢 | 檢索錯誤、文件過期、權限切分錯誤 |
| 函式／工具呼叫 | 呼叫預先定義的功能或 API | 查訂單、計算、建立工單 | 參數錯誤、越權與未驗證輸出 |
| MCP | 以標準方式連接 AI 應用與外部系統 | 串接資料來源、工具與工作流程 | 伺服器信任、授權與供應鏈風險 |
| AI Agent | 依目標規劃步驟並使用工具 | 多階段研究或流程協作 | 過度代理、錯誤累積與責任不明 |

檢索增強生成（Retrieval-Augmented Generation, RAG）的基本流程是：

> 使用者提問 → 檢索相關文件片段 → 將片段加入上下文 → 依片段生成回答 → 顯示來源供查核

文本切分（Chunking）是 RAG 的重要前處理。切得太小可能破壞語意，切得太大則可能帶入過多無關內容，因此應依文件結構與檢索測試調整，而不是套用固定大小。

模型上下文協定（Model Context Protocol, MCP）目前的官方定位是連接 AI 應用與外部資料、工具和工作流程的開放標準。它與 RAG 並非互斥：RAG 著重補充知識，MCP 著重標準化連接，實際系統可能同時使用兩者。

### 6.12 工具類型與選擇原則

工具名稱與功能會快速變動，但選擇邏輯相對穩定。常見類型包括：

- 文字與知識助理。
- 圖像設計與影像編修工具。
- 語音、音樂及影片生成工具。
- 程式輔助與開發代理工具。
- 文件、簡報與資料分析工具。
- RAG、企業知識庫與向量檢索工具。
- 自動化、工具整合與 AI Agent 平台。

評估工具時，可依下列問題逐步判斷：

1. **輸入是什麼？**文字、圖片、音訊、影片、表格或企業文件？
2. **輸出要什麼？**摘要、圖像、程式碼、報告、回答或自動化動作？
3. **資料能否交給該工具？**是否含個資、機密、未公開資訊或受限制資料？
4. **是否需要來源與更新能力？**若回答必須依企業文件，應評估 RAG 與權限管理。
5. **是否要執行動作？**若要串接系統，需評估 API、工具呼叫、MCP 或自動化流程。
6. **結果的風險多高？**錯誤是否會造成對外承諾、財務、健康、安全或權益影響？
7. **能否稽核與維護？**是否可管理權限、版本、紀錄、成本與供應商變更？

特定商業工具不應被寫成永久標準答案。考題與實務選型的重點，是工具類型是否符合任務、資料與風險條件。

### 6.13 實際應用流程

以「製作內部教育訓練教材」為例，可採以下流程：

1. **定義任務**：確認讀者、目的、範圍與完成標準。
2. **準備資料**：整理已核准的制度文件與課程資料，移除不必要的敏感資訊。
3. **選擇工具**：文字內容用文字生成工具；需要內部依據時加入 RAG；插圖再使用圖像工具。
4. **設計提示**：指定角色、任務、來源、格式、限制與查核清單。
5. **先產生架構**：先審查大綱，再分段產生內容，避免一次生成過多而失焦。
6. **逐步修正**：依讀者程度、用語與案例調整。
7. **品質查核**：核對原始文件、數字、引用、圖片與授權。
8. **人工核准**：由內容負責人確認後發布。
9. **保存與更新**：保留使用資料、版本及審核紀錄，來源變更時重新檢查。

### 6.14 生成內容的品質檢查

| 檢查面向 | 核心問題 | 建議方法 |
| --- | --- | --- |
| 正確性 | 事實、數字與引用是否正確？ | 回查原始資料與第一手來源 |
| 完整性 | 是否遺漏重要條件、例外或步驟？ | 對照需求清單與驗收標準 |
| 一致性 | 前後說法、名稱與格式是否一致？ | 交叉檢查並由人員複核 |
| 適用性 | 是否符合對象、目的與使用情境？ | 由實際使用者試讀或試用 |
| 合規性 | 是否涉及個資、機密、侵權或不當內容？ | 套用組織規範並視需要交由專業人員審查 |
| 可執行性 | 內容能否實際操作與驗收？ | 轉為行動項目、負責人與測試案例 |

讓 AI 「自我檢查」可以協助找出表面矛盾或漏項，但不能取代獨立查證，因為檢查者與產生者可能共享相同錯誤。

### 6.15 主要風險與控制方式

| 風險 | 可能情境 | 基本控制方式 |
| --- | --- | --- |
| 幻覺（Hallucination） | 編造來源、數字或不存在的規則 | 限定來源、RAG、事實查核、無依據時拒答 |
| 偏誤（Bias） | 對不同族群產生刻板或不公平內容 | 多元測試、結果比較、人工覆核 |
| 著作權與授權 | 使用未確認權利的文字、圖像、音樂或程式碼 | 保存來源、檢查授權、必要時法務審查 |
| 隱私 | 將個資或可識別資料輸入未核准工具 | 資料最小化、去識別化、權限與保存控制 |
| 資訊安全 | 提示注入、機密外洩或未審查程式被執行 | 輸入輸出驗證、最小權限、環境隔離、稽核 |
| 深度偽造與冒充 | 合成真實人物影像或聲音造成誤認 | 同意與標示、發布審核、來源驗證 |
| 過度依賴 | 使用者未查證就採用結果 | 教育訓練、查核清單、重要決策保留人員 |
| 責任不明 | 錯誤內容發布後無法追蹤決策 | 明確負責人、審核節點、版本及稽核紀錄 |

### 6.16 人工審核與責任邊界

人工參與（Human-in-the-loop）不是只在最後按下確認鍵，而是在人類能理解影響、取得足夠資訊並有權停止流程的前提下，對重要輸入、輸出與動作進行判斷。

下列情境通常需要較嚴格的人工審核：

- 對外發布或代表組織立場的內容。
- 涉及健康、安全、法律、財務或個人權益的建議。
- 使用個資、商業機密或受限制資料。
- AI 將執行付款、刪除、核准、寄送或修改正式系統等動作。
- 內容可能影響特定族群或造成歧視。

責任應由組織中的明確角色承擔，不能以「AI 自動產生」作為免除查核、授權或決策責任的理由。

### 本章重點整理

- 生成式 AI 可產生文字、圖像、音訊、影片、程式碼與跨模態內容。
- 應用規劃應從任務、資料、輸出與風險出發，再選擇工具類型。
- Prompt 應包含任務、背景、資料、格式、限制與檢核要求。
- Few-shot 適合格式模仿；任務拆解適合複雜流程；結構化輸出適合系統串接。
- 上下文工程統整提示、資料、檢索、記憶與工具，不等同於微調模型。
- RAG 補充外部知識；工具呼叫執行功能；MCP 提供標準連接；Agent 負責多步驟任務。
- Temperature 調整隨機程度，不能修補錯誤知識或取代查證。
- 內容品質應檢查正確、完整、一致、適用、合規與可執行性。
- 幻覺、偏誤、著作權、隱私、資安與過度依賴都需要控制措施。
- 高風險內容或動作必須保留有效的人工審核與清楚責任邊界。

### 關鍵名詞

| 名詞 | 簡短說明 |
| --- | --- |
| 生成式 AI（Generative AI） | 依輸入產生新文字、影像、音訊、影片或程式碼的 AI。 |
| 多模態 AI（Multimodal AI） | 能處理或產生兩種以上資料模態的 AI。 |
| Prompt | 提供給生成式 AI 的任務指令、背景、資料與輸出要求。 |
| Prompt Engineering | 設計與優化提示詞，使輸出更符合任務的方法。 |
| Context Engineering | 安排提示、資料、檢索、記憶與工具等整體輸入環境。 |
| Few-shot Prompting | 提供少量範例，讓模型模仿格式或風格。 |
| Structured Output | 要求模型依固定欄位、JSON 或表格等結構輸出。 |
| Temperature | 影響生成結果隨機程度的參數。 |
| RAG | 先檢索外部資料，再依檢索內容生成回答的方法。 |
| Chunking | 將文件切成適合檢索的片段。 |
| MCP | 連接 AI 應用與外部資料、工具及工作流程的開放標準。 |
| AI Agent | 能依目標規劃步驟、使用工具並執行任務的 AI 系統。 |
| Hallucination | AI 產生看似合理但沒有正確依據的內容。 |
| Human-in-the-loop | 在重要流程中保留具實質判斷與控制能力的人員。 |

### 學習重點

1. 文字、圖像、語音、影片與程式碼工具的選擇，先看輸入、輸出與任務，不背特定品牌。
2. 多模態任務必須處理不同模態及其對應關係，不只是把多種檔案一起輸入。
3. Prompt 要說清楚任務、資料、格式與限制；來源不足時應允許模型回答無法判斷。
4. Few-shot 是提供範例；任務拆解是把工作分段；結構化輸出是讓結果可被穩定處理。
5. RAG 解決外部知識檢索問題，MCP 解決標準化連接問題，兩者可以共同使用。
6. Temperature 控制隨機性，不等於事實正確度。
7. 生成內容必須回查來源；AI 的自我檢查不能取代獨立人工驗證。
8. 自動執行的權限越大，越需要最小權限、輸入輸出驗證、人工確認與稽核。
9. 高風險專業判斷應由具資格與責任的人員作最後決定。

### 自我測驗

**1. 某公司要把最新內部 SOP 建成問答系統，回答必須附上文件來源。最適合優先採用哪種方式？**

A. 只提高 Temperature  
B. 使用 RAG 檢索核准文件後再生成回答  
C. 只使用圖像生成工具  
D. 將所有回答固定寫入模型參數

**2. 行銷部門希望 AI 依公司既有範例產生固定格式的產品貼文。哪種提示策略最直接？**

A. Few-shot Prompting  
B. 完全不提供任務  
C. 只增加輸出隨機性  
D. 將資料切成向量但不給任何範例

**3. 下列哪一項最能說明多模態 AI？**

A. 只能把文字改成另一種字型  
B. 同時結合圖片內容與文字問題產生回答  
C. 只使用多個文字段落  
D. 只將同一個問題重複八次

**4. 企業希望 AI 查詢訂單後直接建立退貨申請。除了知識回答外，最需要評估哪項能力與控制？**

A. 工具／API 呼叫，以及授權與人工確認  
B. 只調高圖像解析度  
C. 只增加創作文風  
D. 只做文字分詞

**5. 某生成內容語句流暢，卻引用不存在的統計資料。這最接近哪一種風險？**

A. 幻覺  
B. 吞吐量不足  
C. 資料正規化  
D. 監督式分類

**6. 關於 Temperature，下列敘述何者正確？**

A. 調低後一定能取得最新知識  
B. 調高後一定不會產生偏誤  
C. 它主要影響輸出隨機程度，不能取代事實查核  
D. 它等同於重新訓練模型

**7. AI 產生一段可執行程式碼後，最適當的下一步為何？**

A. 直接在正式環境以最高權限執行  
B. 檢查邏輯與安全性，執行測試並完成人工程式碼審查  
C. 因為可以編譯，所以不需測試  
D. 刪除所有需求文件

**8. 下列哪一項最符合生成式 AI 的責任邊界？**

A. 對外法律結論可由 AI 自動發布，組織不需負責  
B. 重要內容只要由 AI 自我檢查即可  
C. 高風險輸出應由具責任的人員查核、核准並保留紀錄  
D. 工具越自動化，越不需要權限控管

#### 答案與解析

1. **B**。RAG 能先檢索核准且可更新的內部文件，再依文件產生回答並提供來源；Temperature 無法補充企業知識。
2. **A**。Few-shot Prompting 透過少量高品質範例，讓模型模仿既有格式與風格。
3. **B**。多模態 AI 能同時處理兩種以上的資料形式，例如理解圖片後依文字問題回答。
4. **A**。建立退貨申請涉及外部動作，除工具或 API 呼叫外，還必須設計授權、參數驗證、人工確認與稽核。
5. **A**。內容看似合理卻編造不存在的資料，是生成式 AI 幻覺的典型表現。
6. **C**。Temperature 主要影響生成的隨機與多樣程度，不會自動更新知識，也不能保證事實正確。
7. **B**。生成的程式碼仍可能含錯誤、漏洞或不適用相依性，必須經測試、安全檢查與負責任的人員審查。
8. **C**。高風險內容需要有效人工覆核、明確核准者與可追蹤紀錄，不能把責任轉嫁給 AI。
